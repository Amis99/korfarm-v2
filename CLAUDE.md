# 프로젝트 규칙

## 언어
- 모든 응답은 **한국어**로 작성한다. 영어 금지.
- 코드 내 주석, 커밋 메시지 등도 한국어 우선.

## 프로젝트 개요
- 주요 기술 스택: Spring Boot(Kotlin, Gradle) 백엔드, React(Vite, JSX) 프론트엔드, MySQL 데이터베이스
- 한국어 콘텐츠 — 모든 파일 작업 및 bash 명령에서 인코딩을 주의해서 처리할 것
- 이 프로젝트는 snake_case(백엔드/API)와 camelCase(프론트엔드)를 모두 사용함. 필드 네이밍 규칙을 가정하기 전에 항상 실제 API 응답 형식을 확인할 것. API 경계에서 필드를 명시적으로 매핑할 것.

## 디버깅
- 버그 수정 시, 수정을 시작하기 전에 관련된 모든 이슈(예: 누락된 모든 DB 컬럼, 영향받는 모든 테이블)를 한 번에 식별할 것. 에러 발생 시마다 하나씩 컬럼을 수정하는 방식 금지.
- 버그 진단 전에 먼저 데이터 레이어를 확인할 것: DB에 예상 데이터가 있는지, 테이블/컬럼이 존재하는지, 피처 플래그가 활성화되어 있는지 검증. 데이터/설정 이슈를 배제하기 전에 코드 버그라고 가정하지 말 것.
- DB 관련 변경 구현 전: 관련된 모든 테이블의 스키마를 확인하고, 실제 DB 스키마를 JPA 엔티티 정의와 비교하고, 모든 불일치를 하나의 보고서로 나열한 후 수정할 것.

## 코드 변경
- 특정 UI 이슈 수정을 요청받으면, 최소한의 타겟 변경만 할 것. 주변 코드를 리팩토링하거나, 테마 색상을 광범위하게 변경하거나, 명시적으로 요청하지 않은 '개선'을 하지 말 것. 더 넓은 변경이 유익해 보이면 먼저 질문할 것.
- 구현을 요청받으면, 먼저 코드베이스의 현재 상태를 확인할 것. 확인 없이 기능이 이미 구현되어 있다고 주장하지 말 것. 불확실하면 응답하기 전에 관련 파일을 읽을 것.

## 워크플로우
- 기능 구현 또는 수정 후, 커밋하기 전에 반드시 `npm run build`(프론트엔드) 또는 해당 빌드/컴파일 명령어(백엔드)를 실행하여 빌드 에러가 없는지 확인할 것. 테스트되지 않은 코드를 커밋하지 말 것.
- 기능 요청 시: 1) 관련 파일을 모두 탐색, 2) 구체적인 수정사항과 함께 변경할 모든 파일을 나열하는 구현 계획 작성, 3) 사용자가 '승인' 또는 '진행'이라고 할 때까지 구현을 시작하지 말 것.

## Git
- git 작업 시: 의도하지 않은 파일의 커밋을 방지하기 위해 `git add .` 대신 항상 특정 파일 경로로 `git add`를 사용할 것. 사용자가 특정 파일 제외를 요청하면 선택적 스테이징을 사용할 것.
